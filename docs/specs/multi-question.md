# 需求：AskHuman 一次提出多个问题（`-q` 多问题）

> 状态：已确认（待按计划实现）
> 关联计划：`docs/plans/multi-question.md`

## 1. 背景

当前 `AskHuman` 一次调用只能提出**一个**问题（一个位置参数 message，配 `-o` 选项、`-f` 附件、`--no-markdown`）。本需求让一次调用可以提出**多个**问题：弹窗中逐题展示并可前后翻看，最后一次性提交；Telegram 则逐题串行收集；全部答完后才向调用方回传聚合结果。

## 2. 目标

```bash
# 第一题用位置参数（与现有单问题写法完全兼容），其余用 -q
AskHuman "要继续部署吗？" -o "继续" -o "停止" \
  -q "需要保留日志吗？" -o "保留" -o "清除" \
  -q "还有补充吗？" -f ./diff.patch
```

- 新增 `-q` / `--question <text>`，每个 `-q` 开启一个新问题。
- 第一个问题可用位置参数**或** `-q`；**后续问题必须用 `-q`**。
- `-o`（选项）与 `-f`（附件）归属于「最近声明的那个问题」（其前面的位置参数或 `-q`）。
- 弹窗：一次只展示一题，标题右侧显示 `[x/n]` 计数；底部有「上一个 / 下一个」；默认无提交按钮，**所有问题都被查看过后**才在右下角出现「提交」，一次性提交全部回答。
- Telegram：一题一题发送，前一题答完再发下一题；**全部答完才**对调用方返回。
- 弹窗与 Telegram 同时启用时，**哪一端先答完全部问题，就采用哪端整套结果**，另一端立即收尾（抢答以「整个会话」为粒度）。

## 3. 已确认决策

### 3.1 CLI 解析

| 编号 | 决策项 | 结论 |
|---|---|---|
| C1 | 新增参数 | `-q` / `--question <text>`，每次出现开启一个新问题 |
| C2 | 第一题形式 | 第一题可为位置参数或 `-q`；后续题必须 `-q` |
| C3 | `-o`/`-f` 归属 | 归属于「最近声明的问题」（其前的位置参数或 `-q`）|
| C4 | `--no-markdown` 作用域 | **全局**：出现一次即对所有问题生效（不随题） |
| C5 | 位置参数+`-q` | `AskHuman "Q1" -q "Q2"` → Q1 为第 1 题、Q2 为第 2 题；故现有单问题写法天然兼容 |
| C6 | 报错（stderr + exit 1） | ① `-o`/`-f` 出现在任何问题之前；② 位置参数出现在某个 `-q` 之后（位置参数只能是最前的第 1 题）；③ `-q` 缺少值；④ 没有任何问题；⑤ 维持现有：未知选项、`-o`/`-f` 缺值 |

### 3.2 输出契约（stdout）

| 场景 | stdout 输出 |
|---|---|
| 单问题 · 有回答 | **现状格式**（无 `# Q1` 头），原有区块（`[选择的选项]`/`[用户输入]`/`[图片]`/`[文件]`）|
| 单问题 · 空回答 | `[状态]\n用户未回答此问题`（无头；**取代**旧的「用户确认继续」）|
| 多问题 · 全部有回答 | 每题 `# Qn` 头 + 其区块，题与题之间用 `---` 分隔 |
| 多问题 · 部分未答 | 已答题正常输出其区块；未答题输出 `# Qn` + `[状态]\n用户未回答此问题` |
| 多问题 · 全部未答 | **只输出一次**现有取消提示（`[状态]\n用户取消了操作，你必须重新询问用户是否确定要取消，直到用户给出明确答复`）|
| 用户主动取消（弹窗 ⌘W / 关窗，确认后）| 只输出一次现有取消提示 |

约定：

- 「空回答」= 该题没勾选任何选项、没输入文字、没图片、没回复文件。
- 表头用 `# Q1`、`# Q2`…（英文 `Q` + 序号，markdown 一级标题写法），**不回显问题原文**。
- 「全部未答 → 取消提示」**只在多问题触发**；单问题空回答输出「用户未回答此问题」。
- 区块内部格式（含图片落盘路径、文件透传）与现状一致。

### 3.3 弹窗 UI

| 编号 | 决策项 | 结论 |
|---|---|---|
| U1 | 计数显示 | 标题「Question from {名称}」**右侧**显示 `[x/n]`；单问题(n=1)不显示 |
| U2 | 逐题展示 | 一次只展示当前题：正文 + 该题 `-f` 附件 + 该题选项 + 回复输入框/图片/回复文件 |
| U3 | 每题独立状态 | 选项勾选 / 输入文字 / 粘贴图片 / 拖入回复文件 都按「当前题」分别保存，前后切换各自保留 |
| U4 | 导航 | 底部有「上一个 / 下一个」；首题禁用「上一个」、末题禁用「下一个」；单问题不显示导航 |
| U5 | 提交出现时机 | **所有问题都被「查看过」**（切换显示过即算）后，右下角才出现「提交」；出现后在任意一页均可提交 |
| U6 | 底部布局 | 取消移到**底部最左**；「添加图片」从底部移到**输入框正下方**；「上一个/下一个」最初占据右下角（提交的最终位置），**提交出现时把它们往左顶、提交取代「下一个」最右位置**（最终右侧顺序：上一个 · 下一个 · 提交）|
| U7 | 单问题底部 | n=1 时底部回到现状（添加图片 + 取消 + 发送），无导航、无计数、无独立提交逻辑 |
| U8 | 键盘 | `⌘↵`：未全部查看时＝「下一个」，全部查看后＝「提交」；`⌘W`：取消整个会话 |
| U9 | 取消确认 | 若已有部分回答，点取消 / `⌘W` / 关窗时弹二次确认（确定取消则丢弃全部回答并输出取消提示；可继续作答）；无任何回答则直接取消 |

### 3.4 Telegram

| 编号 | 决策项 | 结论 |
|---|---|---|
| T1 | 逐题串行 | 每题依次「发送选项消息 + 操作消息 + 该题 `-f` 文件」，等用户点「↗️发送」后再发下一题；全部答完才回传 |
| T2 | 不可回退 | 线性流程，无「上一个」 |
| T3 | 头部计数 | 每题头部 `「Question from {名称}」[i/n]`（单问题不显示 `[i/n]`）|
| T4 | 无取消 | 维持现状：Telegram 只有「发送」、无取消；某题未选/未输入直接「发送」→ 该题输出 `用户未回答此问题` |

### 3.5 抢答

| 编号 | 决策项 | 结论 |
|---|---|---|
| R1 | 抢答粒度 | 以「整个会话」为粒度：哪一端先答完全部问题，就采用哪端整套聚合结果，另一端立即收尾（不逐题抢答）|

## 4. 约束与既有规则（不可破坏）

- **stdout 洁净**：除新增的 `# Qn` 分组头与 `---` 分隔外，区块语义、图片落盘、文件透传不变；单问题输出与旧版逐字一致（除空回答语义改为「用户未回答此问题」）。
- **向后兼容**：现有单问题写法（位置参数 + `-o`/`-f`/`--no-markdown`）行为不变（视为 1 题）。
- **不引入新 Tauri 插件**；不回退 release 构建的 `tauri/custom-protocol` 修复。
- **现有契约保持**：多 channel 抢答、回答区图片上传、配置文件、`--settings`/`--help`/`--version`、退出码语义（0/1/3）均不变。

## 5. 验收标准

1. `AskHuman "Q1" -o A -q "Q2" -o B` 弹窗显示标题右侧 `[1/2]`，逐题展示，前后翻看各自保留作答。
2. 全部问题查看过后右下角出现「提交」；提交后 stdout 按 §3.2 多问题格式输出（`# Qn` + `---`）。
3. 部分未答 → 未答题输出 `用户未回答此问题`；全部未答 → 单次取消提示。
4. 单问题写法行为与现状一致（无 `# Q1` 头、无计数、无导航）；单问题空回答输出「用户未回答此问题」。
5. CLI 报错场景（§3.1 C6）均 stderr 报错 + exit 1，不弹窗。
6. Telegram（如配置）逐题串行收集，头部含 `[i/n]`，全部答完才回传；与弹窗同启时整会话抢答生效。
7. `--help`、`--agent-help`、参考提示词、README 均反映 `-q` 用法与多问题输出格式。
