# IM 在世即响应（任何消息都给有意义的回复）

## 背景与目标

当 daemon 在运行时，用户在任一**已启用 IM 渠道**发的**任何消息**，都应收到一条**有意义**的回复。
用户据此即可判断「工具在运行（消息已被收到）」，并与「**守护进程已退出**（任何消息都收不到回复）」清晰区分。

硬约束：**完整保留**现有 IM 能力——作答 / 互动卡片 / 抢答 / 自动激活（活跃槽 / 补推）/ 内置命令 / 自动识别 ID，
一律不得回退或改变其最终语义。

## 术语

- **观察者 / 入站监听**：daemon 级 `handle_inbound`（经 `ensure_inbound_listeners` + `spawn_listener` 起的通用循环）。当前只看「发送者 + 文本」。
- **渠道会话**：`ask_question`（卡片模式）/ `ask_question_text`（文本兜底模式）。能看到**完整消息**（含图片 / 文件 / 卡片提交）。
- **卡片模式（主）**：卡片自带「选项按钮 + 文字输入框 + 提交键」；作答期间聊天里的**图片/文件**累积进本题答案，点**提交**才定稿；聊天里的**纯文字不当答案**（答案文字取自卡片输入框）。
- **文本兜底模式**：卡片投放失败时回退；用户回**一条**消息（编号 / 文字 / 图片 / 文件）即完成该题。
- **严格模式（`select_only`）**：忽略自由文字与附件，只认编号 / 勾选。
- **双路由**：每条文本同时分发给「观察者」与「当前活动卡片（当答案）」；但 `/` 前缀文本在 armed（有观察者）时**只**给观察者、不进卡片（视作命令）。图片/文件只有渠道会话能看到。

## 需求（已与用户逐条确认）

### R1 存活即监听
- 只要 **daemon 存活 + 配置中有启用的 IM**，就保持监听；**与「工作中 agent / 是否有在途提问 / 自动激活开关」全部无关**。
- daemon **启动时**即在**后台**连启用的 IM 并起监听（后台执行，**不阻塞弹窗关键路径**，不回退既有启动性能优化）；配置变更后按新配置重连。
- 监听**贯穿 daemon 整个生命周期**，daemon 退出才断。
- **不改变空闲退出**：监听本身**不**阻止 daemon 空闲自动退出（无在途请求 / 无窗口 / 无工作 agent 时照常退出）。→ 「能回复＝在运行；静默＝已退出」成立。
- 取舍（用户已知悉）：启用了 IM，则 daemon 在世期间常驻一条到该 IM 的连接（长轮询 / WebSocket），并在启动时后台读一次钥匙串取密钥。

### R2 作答确认（内容被接受进答案时）
- **仅当**消息内容**确实被接受进本题答案**时，回一条确认：「✅ 已收到，将把该 **{图片/文件/内容}** 加入你的回答」；卡片模式追加「请在卡片点**提交**完成」。
- 判定依「渠道 / 模式能力矩阵」（见计划，逐家审计后写死）：
  - **卡片模式**：聊天里的**图片 / 文件**被累积进答案 → 确认；**纯文字**不当答案 → 归 R3。
  - **文本兜底模式**：被接受的**文字 / 编号 / 图片 / 文件** → 确认。
  - **严格模式**：被忽略的自由文字 / 附件 → 归 R3。
- 同一题作答期间连发多条附件：**每条都回一条确认**（即时反馈）。
- 图片 / 文件确认由**渠道会话**负责（只有它能看到附件）。

### R3 引导文案（消息未被当作答案时）
- 对**未被当作答案、也不是已识别命令**的消息，回一条**动态引导文案**；**不写「已收到」**（能回复本身即代表收到且在运行）。
- 适用场景：卡片模式下的纯文字、严格模式被忽略的内容、**当前无在途提问**时的任意消息、**未知 `/命令`**。
- 文案按开关**动态分块**拼装：
  - **可用命令**：`/status`、`/help`；`/here` **仅自动激活开**时列出。
  - **如何作答**（仅当前有在途提问时）：卡片模式「在卡片内选择 / 输入后**提交**，可发图片 / 文件补充」；文本模式「直接回文字 / 编号 / 图片」。
  - **自动激活开**时：补「发任意文字＝把提问切到此渠道接收」。
  - **当前无在途提问**时：「暂无进行中的提问」。
- 新增显式 **`/help`** 命令，复用同一文案。

### R4 既有命令不变
- `/status`：维持现状（**始终**回状态文本，与开关无关；自动激活开且因此切槽时附激活回执）。
- `/here`：自动激活**开**时维持现状（激活 + 补推 + 回执）。自动激活**关**时，由现状的「静默忽略」改为**回引导文案**（归未知 / 无效命令）。

### R5 自动识别 ID 成功回执
- 自动识别（用户发 4 位识别码）**成功**后，经该 IM 回一条确认：「✅ 识别成功，已自动填入 **{字段名}**」，**不回显完整 ID**（隐私）。
- 字段名按渠道：钉钉 `userId` / 飞书 `openId` / Slack `userId`（沿用各自「测试连接 / 识别」既有字段命名）。
- 失败 / 超时 / 取消：维持现状，不回执。

## 协调原则（避免重复回复，单消息至多一条回复）

- **命令（`/` 前缀）**：路由层 armed 时不下发给卡片会话 → **始终由观察者处理**（已识别命令各自处理；未知命令 → 引导）。
- **图片 / 文件**：仅渠道会话可见 → 确认（R2）/ 引导（R3 严格模式）由**会话**负责。
- **非命令文本**：路由层同时给会话与观察者 → 由「**该渠道当前是否有活动在途提问**」决定归属：
  - **有** → 会话负责（接受则确认、不接受则引导），观察者**退避**（对该文本不再处理）。
  - **无** → 观察者负责（自动激活开则按现状切槽 / 回执；否则 / 未切换则回引导）。
- **非文本消息（图片）且当前无活动在途提问**：经 R1 放宽后的监听让观察者也能感知「来自期望发送者的非文本消息」→ 回引导（实现见计划）。

## 不做 / 边界

- 监听**不**阻止空闲退出。
- **不改变**作答最终语义（答案内容、提交、抢答、补推一律不变）。
- 不引入需要公网的新依赖。
- 不改变弹窗（popup）渠道行为（无 IM 收件端）。
